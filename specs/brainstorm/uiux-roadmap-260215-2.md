>
> 由 Proposer（方案提出）、Researcher（業界研究）、Devil's Advocate（批判審視）三方意見整合

## 執行摘要

第二輪腦力激盪在第一輪的基礎上，探索了全新的 UI/UX 改進方向。Proposer 提出 8 個提案、Researcher 提供 9 個業界建議，經 Devil's Advocate 嚴格批判後，再經使用者篩選，最終精煉為 **2 個執行項目** + **3 個待驗證項目**。

**Devil's Advocate 核心提醒**：這一輪提案過度關注「狀態可見性」（加指標、進度條），但 Auto-Sub 的核心是「即時翻譯字幕」，不是「系統監控儀表板」。在沒有使用者回饋數據前，應聚焦把核心體驗做穩做好。

## Tier 1：建議執行（高 ROI、低風險）

### 1.1 翻譯中狀態 Spinner（縮減版）

**問題**：字幕顯示「翻譯中...」文字時，使用者不確定系統是否卡住。

**批判後調整**：
- ~~閃爍點表示正在聆聽~~ → 不做，interim 文字本身已有回饋
- ~~Menu Bar 即時文字預覽~~ → 不做，違反 HIG
- ~~上下文修正顏色漸變動畫~~ → 不做，視覺噪音

**保留的改進**：
1. **「翻譯中...」文字旁加入小型 ProgressView（spinner）**：讓使用者知道系統正在工作
2. 翻譯完成後 spinner 自然消失

**相關檔案**：`SubtitleOverlay.swift:163-171`（translatingText 顯示處）
**難度**：低
**依據**：Researcher 語音辨識 UI 狀態回饋研究

---

### 1.2 Profile 選單資訊增強（縮減版）

**問題**：Menu Bar 的 Profile 子選單只顯示名稱，使用者無法快速辨識各 Profile 的用途。

**批判後調整**：
- ~~懸浮預覽~~ → 不做，工程浪費
- ~~快捷鍵循環切換~~ → 不做，5+ 個 Profile 的快捷鍵設計不合理
- ~~Phase 2 Popover~~ → 不做，維護成本高

**保留的改進**：
1. **Profile 選單項顯示語言對 + 場景標籤**：格式如「日劇場景 (ja→zh-TW) ✓」
2. 選中的 Profile 用 ✓ 標記

**相關檔案**：`MenuBarController.swift:259-271`
**難度**：低
**依據**：Proposer codebase 分析 — 選單資訊不足導致辨識困難

## Tier 2：待驗證（需使用者回饋確認痛點）

### 2.1 翻譯失敗自動重試

**Proposer 提案**：翻譯失敗時提供手動重試按鈕。
**Devil's Advocate 批判**：應該自動重試而非讓使用者點按鈕；且失敗率未知，可能是罕見情況。

**如果要做**：
- 改為 **自動重試**（backoff + 最多 3 次），不加按鈕
- 3 次皆失敗才顯示「翻譯失敗（已重試 3 次）」
- **前提**：先收集翻譯失敗率數據，確認 > 5% 才值得投入

**相關檔案**：`main.py:165-180`、`translator.py:186-250`

---

### 2.2 多螢幕位置記憶

**Proposer 提案**：以螢幕 UUID 為 key 儲存不同螢幕的字幕位置。
**Devil's Advocate 批判**：使用場景不明確，UUID 儲存過於精細。

**如果要做**：
- 簡化為記住「主螢幕」和「副螢幕」的相對位置（如「右下角 20% 邊距」）
- 啟動時檢查位置是否超出螢幕邊界，超出則重置到預設位置
- **前提**：先確認使用者是否真的在多螢幕間頻繁切換

**相關檔案**：`AppState.swift:79-87`、`SubtitleWindowController.swift:279-290`

---

### 2.3 設定參數 Preset

**Proposer 提案**：為 Deepgram 參數加入場景說明和快速設定 Preset。
**Devil's Advocate 批判**：應簡化參數而非加更多說明。

**如果要做**：
- 只保留 **Preset 按鈕組**：「快節奏」「標準」「穩定」三個預設
- 點擊後一鍵設定 endpointing/utterance_end/max_buffer_chars 組合
- 考慮將技術參數預設隱藏在「進階」摺疊區
- **前提**：先確認使用者是否真的會調整這些參數

**相關檔案**：`ProfileSettingsView.swift:335-359`

## Tier 3：推遲（低 ROI 或過度設計）

| 原提案 | 推遲原因 |
|--------|---------|
| P0-1 字幕捲動狀態視覺化 | 假設痛點無數據支持；偏離核心價值 |
| P1-3 擷取統計顯示 | 開發者指標 ≠ 使用者需求；上次已批判過 |
| P2-1 翻譯品質視覺回饋 | 技術細節暴露；應改進翻譯本身而非加指標 |
| P2-2 錯誤恢復進度視覺化 | 治標不治本；錯誤恢復應透明 |
| R: 顏色對比度驗證 | 使用者判斷目前可讀性已足夠 |
| R: 字幕淡入動畫 | 使用者判斷非必要 |
| R: Menu Bar 即時文字預覽 | 違反 HIG 精神；視覺噪音 |
| R: 上下文修正顏色漸變 | 視覺噪音，現有 mint 色已足夠 |
| R: 翻譯品質指標（信心度） | API 不一定支援；對使用者意義不明 |
| R: 字幕預設風格 | 設定膨脹，第一輪已批判過 |
| R: 字幕匯出 SRT/VTT | 嚴重偏離 MVP，第一輪已批判過 |
| R: Window Style MenuBarExtra | 重構成本高，收益不明確 |

## 建議執行順序

```
Phase 1（0.5 天，低難度）
├── 1.1 翻譯中 Spinner（小型 ProgressView）
└── 1.2 Profile 選單資訊增強（語言對 + 場景標籤）

Phase 2（收集使用者回饋後決定）
├── 2.1 翻譯失敗自動重試（需先確認失敗率）
├── 2.2 多螢幕位置記憶（需先確認使用者需求）
└── 2.3 設定參數 Preset（需先確認調參需求）
```
